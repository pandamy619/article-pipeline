# Роадмеп: картинки к постам

Цель — в редакторе поста («Правка») можно заменить «стандартную» картинку
(`og:image` из источника) на свою: **загрузить файл**, **найти в интернете**
(сток/SearXNG) или **сгенерировать локально** (ComfyUI на RTX 4080). Картинка
пишется в существующее поле `articles.image_url` и уходит в публикацию.

## Ключевые архитектурные решения (до старта)

1. **Хранилище своих/сгенерированных картинок.** Сейчас `image_url` — это внешний
   URL источника. Для загруженных и сгенерированных нужен свой стор: каталог в
   Docker-томе (`media/`) + раздача статикой (через nginx `web` или эндпоинт
   admin). `image_url` тогда указывает на наш `/media/<file>`.

2. **ВАЖНО: публикация локальных картинок.** Сейчас бот постит `image_url` как
   **ссылку** — Telegram сам её скачивает, значит URL должен быть **публичным**.
   Для локальных файлов (`/media/...`, недоступно из интернета) так не выйдет:
   бот должен слать **сами байты файла** (`FSInputFile`/`BufferedInputFile`), а не
   URL. → `app/src/publisher/telegram.py` учит различать «внешний URL» и
   «локальный файл» и грузить файл напрямую. Это фундамент для фаз 1 и 3.

3. **Источник картинки.** Опционально добавить `articles.image_source`
   (`source | upload | search | generated`) — для понятности в UI и аналитики.
   Не обязательно для MVP, но полезно.

## Фаза 0 — Фундамент (S)

Без этого остальное не поедет.

- Том `media/` + раздача статикой; маршрут к файлу.
- `telegram.py`: публикация локального файла байтами (решение №2 выше).
- В «Правке» — блок картинки: превью текущей + переключатель источника
  («из источника» / «убрать»). Пока без новых источников.
- Эндпоинт «установить/очистить картинку статьи».

Итог: картинка видна и управляема в редакторе, публикация локальных файлов работает.

## Фаза 1 — Загрузка своей (S–M) ← наибольшая ценность, проще всего

- Backend: `POST /api/articles/{id}/image` (multipart) → валидация типа/размера →
  сохранить в `media/` → проставить `image_url` → вернуть.
- Frontend: кнопка «Загрузить» → выбор файла → загрузка → превью обновляется.
- Зависимостей нет, GPU не нужен. Закрывает основной запрос «своя картинка».

## Фаза 2 — Найти в интернете (M)

- Backend: `GET /api/articles/{id}/image/search?q=...` → сток-API
  (**Pexels/Pixabay**, лицензия под коммерцию) и/или **SearXNG** (категория
  `images`, уже стоит) → вернуть превью+URL. Ключевые слова может предложить LLM
  из текста поста.
- Frontend: «Найти» → поле запроса (предзаполнено LLM) → галерея → выбор.
- Для безопасной публикации лучше **скачивать** выбранную в `media/` (не хотлинк).
- Решение: сток-ключ (Pexels/Pixabay бесплатные) vs SearXNG (охват шире, но
  чужие картинки из выдачи — юридический риск, на усмотрение).

## Фаза 3 — Локальная генерация (L) ← «вау», тяжелее всего

- Инфра: контейнер **ComfyUI** с GPU (отдельный профиль), workflow-JSON под модель.
  Модель: **FLUX.1 schnell** (быстрая, Apache-2.0/коммерция) или **SDXL** (легче,
  большая экосистема). FLUX dev — некоммерческая лицензия, осторожно.
- Backend: `POST /api/articles/{id}/image/generate` → LLM строит промпт из поста →
  вызов ComfyUI API → ожидание → сохранить результат в `media/` → `image_url`.
  Делать **асинхронно через очередь задач** (как ручной сбор `CollectJob`), чтобы
  не блокировать запрос и не ловить таймаут.
- Frontend: «Сгенерировать» → промпт (предзаполнен) → прогресс → превью.
- **Дележка VRAM:** 4080 одна, 16 ГБ. FLUX dev (~12 ГБ) одновременно с Ollama не
  влезет → выгружать LLM на время генерации или генерить, когда LLM простаивает.
  SDXL (~7 ГБ) уживается легче. Стратегию выбрать заранее.

## Что нужно решить перед стартом фаз

- **Фаза 2:** сток-API (Pexels и/или Pixabay) — завести ключи; включать ли
  SearXNG-картинки несмотря на копирайт.
- **Фаза 3:** модель (FLUX schnell vs SDXL) под лицензию; стратегия VRAM
  (выгрузка LLM vs очередь «когда свободно»).

## Рекомендуемый порядок

1. **Фаза 0 + Фаза 1** вместе — загрузка своей картинки + публикация байтами.
   Это уже полностью закрывает «заменить стандартную на свою».
2. **Фаза 2** — авто-подбор релевантного фото без GPU.
3. **Фаза 3** — локальная генерация, когда захочется уникальных картинок.

Каждую фазу — отдельным набором коммитов, с тестами и проверкой публикации
(локальный файл реально уходит в Telegram).
